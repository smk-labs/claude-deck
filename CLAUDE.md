# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Never run the patch without asking

**Do not run `claude-deck patch`, `claude-deck revert`, or anything that touches `/Applications/Claude.app` unless the user explicitly asks for it in this session, and only after confirming Claude Desktop is quit.** Smoke-testing the patch quits the running app: `patch` and `revert` both need Claude closed to rewrite `app.asar`, and the script will try to quit it via `osascript`. If Claude is open with unsaved context (an in-progress chat, a login flow), killing it is disruptive. Ask first, every time. `status` and `list` are read-only and always safe to run.

## Never quit or kill a running Claude process yourself

**Do not run `claude-deck stop`/`kill`, `quit_claude`, or any raw `kill`/`pkill` against a real Claude.app process — not even one instance, not even to verify a fix — under any circumstances, full stop.** This is a standing rule, not a per-session ask: if something needs verifying against a live, running Claude window, describe what to check and have the user run it themselves. This is stricter than the patch/revert rule above (which is "ask every time"): for quitting Claude, the answer is always no unless the user reopens the topic. The dashboard's own Node server (port 8965) and scratch `--app` test targets are NOT covered by this — those are fine to start/stop freely, they carry no account or session state.

## What this repo is

A Bash script (`claude-deck.sh`, macOS) and its PowerShell twin (`claude-deck.ps1`, Windows), plus a small local dashboard (`dashboard/`). Each script patches the Claude Desktop app on its platform to accept a `--profile=NAME` launch argument, so each profile gets its own Electron `userData` directory, its own login session, and can run side by side with the others. The dashboard is a zero-dependency Node HTTP server that reads each profile's session key and shows its usage limits. There is no build system, no package manager, no tests: the two scripts and the two dashboard files *are* the product. An optional Cursor integration (`dashboard/cursor.js` plus `integrations/claude-code/`) sits alongside, off by default and untouched by the patch flow: see "Cursor integration" below.

## Running it

```bash
./claude-deck.sh patch              # apply (idempotent)
./claude-deck.sh patch --force      # re-apply even if already patched
./claude-deck.sh patch --verify-launch  # (only with --app <scratch-copy>) launch + confirm it stays alive 8s
./claude-deck.sh revert             # restore original Claude.app from backup
./claude-deck.sh status             # show patch state, hashes, backup info, known profiles
./claude-deck.sh open <name> [org-uuid]  # launch (or focus) a profile, optionally switched to one org first
./claude-deck.sh list                # list known profiles and their cached usage
./claude-deck.sh dash [port]         # start the local dashboard (default port 8965)
./claude-deck.sh stop [port]         # (alias: kill) stop the dashboard + quit every open profile
./claude-deck.sh install             # copy to ~/.claude-deck/ + add zsh alias
./claude-deck.sh uninstall           # remove the alias only
./claude-deck.sh watchdog on|off     # install/remove the LaunchDaemon that re-patches on update (needs sudo)
```

`patch` and `revert` need `sudo` (writes into `/Applications/Claude.app`) and need Claude quit first. `open`, `list`, `dash`, and `status` never touch the app bundle and never need sudo. `stop` needs no sudo either and never touches the app bundle, but it does quit every running Claude instance — see the rule above.

To smoke-test a change: confirm with the user, then edit the script, run `./claude-deck.sh revert` (if previously applied), then `./claude-deck.sh patch`, then `./claude-deck.sh open work` and verify the window title shows `[work]` and the profile gets its own login. `status` is the fastest sanity check between iterations.

## Patch-free launch path (CLAUDE_USER_DATA_DIR)

Plain-language: profile launches no longer need the patch; Claude itself supports a per-launch data dir.

- `cmd_open` (and the no-script fallback in `server.js`) launches named profiles as `CLAUDE_USER_DATA_DIR="<profile dir>" open -n -a Claude --args --profile=<name>`. The env var is Claude's own hook: an unconditional top-level block in `.vite/build/index.pre.js` (verified 1.21459.0, no platform guard, also used by Anthropic's `CLAUDE_CDP_AUTH` machinery) does the same `setPath('userData', ...)` the patch injects, plus `setPath('logs', <dir>/Logs)`; a `Logs/` dir inside the profile dir is the tell that the env block ran. `open` forwards the caller's env (open(1)).
- The `--profile` flag is still passed deliberately: `_profile_is_running`/`getRunningProfiles` detect running profiles by scanning ps args for it, and while the app is patched the injected code needs it for the `[name]` title prefix and the session-key exporter's label. An env-only launch on a *patched* app mislabels itself `default` and clobbers `profiles/default.json` (observed live). On an unpatched app the flag is inert.
- Still patch-only: the `[name]` window-title prefix and the sessionKey exporter (dashboard key freshness). Everything else (separate logins, session-index symlink via the shell twin, running detection, dash) works patch-free. Windows twins (`claude-deck.ps1` `Start-ClaudeInstance` + `server.js` win32 fallback) set the env var the same way; on Windows this is the ONLY launch path, since MSIX retired the patch entirely (see "Windows port" below).

## Org-targeted open (macOS + Windows)

Plain-language: one account can belong to several orgs (Team, Team-2, a personal Max plan). `open <name> <org-uuid>` launches that profile already switched to a specific org, instead of whatever org it last had active. Patch-free on both platforms: no asar, no app bundle, no codesign anywhere in this path.

- The active org lives in a cookie, `lastActiveOrg` on `.claude.ai`, value = org UUID, encrypted Chromium v10. **macOS**: AES-128-CBC, key = PBKDF2-SHA1 of the Keychain password (service `Claude Safe Storage`/account `Claude Key`, 1003 iters, 16 bytes), fixed IV of 16 spaces; one keychain item shared by every profile. **Windows**: `v10` + 12-byte nonce + AES-256-GCM + 16-byte tag, key = DPAPI-unprotect of each profile's own `Local State` `os_crypt.encrypted_key` (per-profile, not shared). Confirmed live on both: seeding this cookie and launching lands the app on that org, from the CLI and the dashboard's real `/api/open`.
- `dashboard/cookie-crypto.js` is the shared, platform-aware helper: one module (`seedOrg`, `decryptV10`, `encryptV10`, `deriveKey`, `UUID_RE`) + a `node cookie-crypto.js seed-org <cookiesDb> <orgUuid>` CLI, branching on `process.platform`. sqlite: macOS shells to the system `sqlite3` CLI; Windows uses `node:sqlite` (node >= 22.5, so the ps1 bootstraps Node 22 — if a caller's node lacks it, `seedOrg` returns `{ ok:false, reason:'no-sqlite' }` and the launch proceeds normally). The plaintext is `[N leading bytes][36-byte org UUID]`; `seedOrg` never fabricates the leading bytes, it decrypts the profile's own existing row, keeps everything before the trailing 36 bytes, and splices the new UUID in after them (macOS N≈34 constant; Windows N=32 = the SHA-256 of the host_key Chromium prepends). A profile with no existing `lastActiveOrg` row can't be seeded and falls through to a normal launch.
- `cmd_open name [org-uuid]` (sh), `Cmd-Open name [org-uuid]` (ps1), and `POST /api/open {name, orgId}` all gate the seed behind the SAME not-running check the launch already uses, via real control flow (the `else` branch of the is-running check), not a separate up-front check: writing to a live Cookies WAL file from outside is externally silent (no crash, no lock error) but the running app can later overwrite or ignore it. **A profile that's already running is always just focused, org untouched, full stop** — switching orgs mid-session would yank the user out of whatever they're doing. Verified: a different-org request against a running profile leaves the Cookies mtime unchanged and returns the same `{ok:true, activated:true}` a plain open gives.
- `server.js` passes `orgId` through as a 2nd CLI arg to whichever script it finds (`claude-deck.sh`/`.ps1`), so the shell/ps1 `open` does the seeding and it's never done twice; it only seeds the cookie itself in the no-script fallback (macOS `Cookies`, Windows `Network\Cookies`).
- Dashboard: only the Up-next org pill passes `orgId` (it's the one row tied to one specific org); the profile card's own Open button still opens with no org, unchanged.

## stop / kill, and the pgrep/pkill trap

Plain-language: `stop` (alias `kill`) is a force-quit-everything command — it stops the dashboard's Node server and quits every open Claude profile, default included. It exists precisely for when quitting windows one by one isn't good enough; see the rule near the top of this file about never running it yourself.

- `cmd_stop [port]`: kills whatever's listening on the dashboard port (`lsof -ti tcp:<port> -sTCP:LISTEN`, default 8965) if anything is, then calls `quit_claude` (see below) to close every running Claude instance across every profile.
- **`pgrep -x "Claude"` / `pkill -x "Claude"` cannot see Claude's main process at all on this hardened-runtime build — confirmed live, and this was a real, silent bug in `quit_claude()` from the start**, not something introduced by `stop`. Two independent, stacked causes: (1) the process's kernel `comm` name is the FULL executable path (`/Applications/Claude.app/Contents/MacOS/Claude`), not the bare `Claude` basename `-x` expects (`ps -o comm` shows this truncated to `/Applications/Cl`); (2) even switching to `-f` (full command line) still finds nothing for the MAIN process specifically — `pgrep -f "Claude"` matches every one of its child Helper/renderer/crashpad processes just fine, but never the top-level one, even though `ps` lists its full command line with no trouble. This is a macOS argv-visibility protection on the main binary, confirmed independent of any sandboxing (reproduced identically with the agent's own sandbox fully disabled). Because of (2), `pgrep -x "Claude"` failing silently meant `quit_claude()` returned before even attempting the AppleScript quit — every previous `patch`/`revert` run had been relying entirely on the user having already quit Claude manually first, exactly as the top-of-file rule already asks for.
- Fix: `_running_claude_main_pids()` uses the same `ps ax -o pid,command | grep -F "Claude.app/Contents/MacOS/Claude"` scan `_profile_is_running` already relies on for detection, and `quit_claude()` signals by PID directly with a plain `kill`/`kill -9` rather than `pkill`. A bare `kill(pid, sig)` only needs same-UID signal-send permission — a completely different, unrelated check from the argv-read permission `pgrep`/`pkill`'s matching syscall lacks for this process. Escalation ladder: AppleScript `tell application "Claude" to quit` first (up to 5s to take effect), then SIGTERM by PID, then SIGKILL by PID if still alive 2s later.

## Architecture / how the patch works

The things that make this work as a self-contained script: touch any of them and you can break the whole flow:

1. **Asar extract → inject → repack.** The Electron app's code lives in `/Applications/Claude.app/Contents/Resources/app.asar`. The script uses `@electron/asar` (installed locally into `~/.claude-deck/tool/`, **pinned to ^3**: v4+ is ESM-only and breaks both the CLI invocation and the inline `require()` in the header-hash step) to extract, write `claude-deck.js` at the asar root, prepend `try { require('../../claude-deck.js') } catch (e) {}` to `.vite/build/index.pre.js` (the Electron main-process entry point), then repack.

1a. **Unpacked native modules must survive the repack, or the app never opens.** Modern Claude Desktop ships native `.node`/`.dylib` files and node-pty's `spawn-helper` marked `unpacked:true` in the asar header (materialized on disk in `Contents/Resources/app.asar.unpacked`), because Electron cannot `dlopen`/`execve` a file that lives inside the asar archive. A plain `asar pack` with no `--unpack` rule silently repacks everything, and the main process crashes before any window opens: this bricked the app once already. `asar_pack_preserving_unpacked()` collects the original unpacked file list, packs with an exact basename brace-glob (`@electron/asar` matches `unpack` against each file's basename, not its full path, so full-path globs match nothing), falls back to a generic `{**/*.node,**/*.dylib,**/spawn-helper}` pattern if that doesn't reproduce the set, and the caller re-verifies set equality against the original before installing anything. `asar.extractAll` also drops the executable bit on unpacked files (they land as plain 644), so `cmd_patch` restores each one's real mode from the live `Resources/app.asar.unpacked` before repacking, or `spawn-helper` (needs `+x` to `execve`) would silently stop working even though the app would still launch.

2. **ElectronAsarIntegrity hash.** Electron refuses to load the asar if the SHA-256 in `Info.plist` (`:ElectronAsarIntegrity:Resources/app.asar:hash`) doesn't match. The hash is computed over the asar **header JSON**, not the whole file. After repack, the script recomputes it and updates the plist via `PlistBuddy`.

3. **Ad-hoc re-sign, never `--deep`, entitlements stripped not preserved.** `codesign --force --sign - --preserve-metadata=identifier,flags,runtime --entitlements <derived.plist>` on the outer bundle only. **Do not add `--deep`**: it re-signs nested helpers as ad-hoc, which invalidates their keychain ACLs and causes repeated keychain prompts on every Claude launch. This is one of two load-bearing constraints in this mechanism; the fix carries over verbatim from the sibling `claude-rtl` project.
   - **Why entitlements are no longer just preserved.** The real install is hardened-runtime and carries Apple *restricted* entitlements (`com.apple.application-identifier`, `com.apple.developer.team-identifier`, `keychain-access-groups`), which AMFI only permits under a genuine Apple team signature. `--preserve-metadata=...,entitlements,...` used to carry those onto the ad-hoc signature verbatim; on macOS 26 (Tahoe) AMFI then refuses to launch the binary at all ("adhoc signed but contains restricted entitlements") even though `codesign --verify` reports the signature as structurally valid. Dropping those keys then breaks Library Validation for the still-genuinely-signed nested Electron Framework ("different Team IDs"), so `com.apple.security.cs.disable-library-validation` is added to compensate. Hardened runtime stays on; `--deep` is still never used (disable-library-validation makes it unnecessary).
   - **`build_adhoc_entitlements()`** derives this plist fresh from the target app's own current entitlements every time (dump via `codesign -d --entitlements`, handling both the direct-file and `:-`-stdout forms; strip the three restricted keys with `PlistBuddy -c Delete`; add the disable-library-validation key), so it self-adapts if Anthropic's entitlement set changes later. Falls back to a hardcoded minimal set (JIT + the device/personal-information/virtualization keys the app is known to use) only if the app is unsigned and the dump comes back empty. Used identically by `cmd_patch`'s success path, its rollback path, and `cmd_revert`: revert can only ever produce an ad-hoc signature too (the pristine asar's original signature was Anthropic-genuine, but ad-hoc can't carry it), so it needs the same stripped set.
   - **`assert_launch_safe_entitlements()`** is a static post-sign gate: dumps the just-signed entitlements back and asserts none of the three restricted keys survived and `disable-library-validation` is `true`. This exists because `codesign --verify` alone is a false-negative here: it validates signature structure, not AMFI's separate launch-time policy check, so it happily passed on a binary AMFI still refused to run. Runs in `cmd_patch` (fails → rollback trap fires) and `cmd_revert` (fails → `die`).

4. **The injection itself** (`claude-deck.js`, written at patch time). Runs in the main process before `app.ready`, wrapped so nothing inside it can crash the app:
   - Reads `--profile=` from `process.argv`, sanitizes to `[A-Za-z0-9_-]`, caps at 32 chars.
   - If a profile is set, calls `app.setPath('userData', ...)` (and best-effort `app.setPath('sessionData', ...)` in its own try/catch) pointing at `~/Library/Application Support/Claude Profiles/<name>`.
   - **Shares one Claude Code session index across profiles.** Claude Desktop keeps its Claude Code session list per-`userData`, at `<userData>/claude-code-sessions/<account-uuid>/<org-uuid>/local_*.json`. Transcripts live in the shared `~/.claude/projects`, but the app only lists sessions found in this per-profile index, so a second profile of the same account showed "no Code sessions" even with transcripts present. Fix: symlink each profile's `claude-code-sessions` dir at the default app's own (`~/Library/Application Support/Claude/claude-code-sessions`), so every profile reads and writes the same index. If a profile already has a real (non-symlink) index dir, its contents are migrated into the shared dir additively (never overwriting a file already there), the original directory is kept as a `.migrated-<timestamp>` backup (never deleted), and only then is the symlink created. Runs once at require time, before `app.on('ready')`, since the fix must be in place before the app first touches the dir. Every step has its own try/catch; any failure leaves the old per-profile behavior in place rather than blocking launch. This is also why the shared `~/.claude` design keeps `claude-sync`'s single-directory scan assumption valid: see "Shared account data, separate accounts" below. The same logic also exists as a shell twin, `ensure_profile_index_link()`, run by `cmd_open` before every profile launch, by `cmd_dash` at startup, and by `cmd_doctor` for all profiles at once. That makes linking self-healing even when the installed app carries an outdated injection (a real failure mode, hit twice: a scoping bug in the injected JS, then an un-repatched app). The injected copy is belt, the shell twin is suspenders; keep them behaviorally identical when touching either.
   - On `app.on('browser-window-created')`, listens for `page-title-updated`, calls `preventDefault()`, and sets the title to `[<profile>] <original title>` so Mission Control, Cmd-backtick, and Raycast can tell windows apart.
   - After `app.whenReady()`, reads the `sessionKey` cookie for `https://claude.ai` and writes it to `~/.claude-deck/profiles/<label>.json` (mode 600), merging into any existing file so a cached `orgId` survives. Re-pulls every 30 minutes and on `ses.cookies.on('changed')` for `sessionKey`. Before that first pull the reverse direction runs (session seeding): if the JSON already holds a `sessionKey` and the session has no `sessionKey` cookie at all, the key is planted as a cookie (secure, httpOnly, lax, 60-day expiry) and any window that had already loaded gets one reload; an existing cookie is never overwritten, and the pull is chained after the seed so it can't re-write the JSON from a stale read. This is the login-import half of profile migration: see "Migrating profiles to another machine" below. Every failure path is silent: this code must never be the reason Claude fails to launch.

5. **Backup + idempotency.** First patch copies the pristine asar (and `app.asar.unpacked`, if present) to `~/.claude-deck/backup/` and stashes the original Info.plist hash + Claude version. The marker `claude-deck.js` inside the asar is the "is patched" signal. `revert` restores the asar, the unpacked dir, and the hash, then re-signs. `patch --force` re-applies even when the marker is already present (useful after editing the injected script).

6. **Preflight gate, rollback trap, post-validation.** `cmd_patch` checks `.vite/build/index.pre.js` exists in the asar *before* touching anything (Claude's internal layout can change between versions); if it's gone, nothing is modified. From the moment the installed app starts changing until success, an `EXIT`/`INT`/`TERM` trap (`_patch_rollback`) is armed: any failure restores the backup asar, the backup unpacked dir, and the plist hash, re-signs, and reports honestly if the restore itself couldn't fully complete (e.g. a read-only `Info.plist`) rather than claiming false success. `PlistBuddy -c "Set ..."` can exit 0 while silently failing to write, so the trap doesn't trust its own exit code either: it re-reads the plist afterward and compares. Before printing success, a post-validation gate re-derives ground truth (asar header hash equals the plist's, the marker is present, the unpacked set still matches, `codesign --verify` passes) and only then clears the trap.

7. **Testability via `--app <path>` / `CLAUDE_DECK_APP`.** Every subcommand accepts a hidden `--app` override (or the env var) that redirects `APP`/`RES`/`ASAR`/`PLIST` at an alternate `.app` bundle, so the whole patch/revert/rollback flow can be smoke-tested against a scratch copy without touching the real `/Applications/Claude.app`. When the target bundle is user-writable, every file mutation and `codesign` call skips `sudo` automatically (`$SUDO` resolves to `""`). Backups for an alternate app live under `~/.claude-deck/backup-alt/`, never `~/.claude-deck/backup/`, so a test run can never read from or clobber the real backup. `quit_claude()` itself is scoped the same way: it's a no-op unless `$APP` is exactly `/Applications/Claude.app`, since `osascript`/`pkill` there target the process by bare name ("Claude") and would otherwise hit the real running app regardless of which bundle is actually being patched. `patch --verify-launch` builds on this: it spawns the freshly-patched `$APP` with a throwaway `--user-data-dir` and confirms the process is still alive 8 seconds later (the one check that exercises the real AMFI launch-time policy end-to-end, vs. `assert_launch_safe_entitlements`'s static proxy for it). It refuses outright, via `die`, if `$APP` is the real Claude.app.

## Windows port (`claude-deck.ps1`)

The Windows twin of the bash script. Same subcommands, same safety architecture (preflight gate before any mutation, version-stamped pristine backup, rollback via try/finally, unpacked-set equality verified before install and again after). What differs, and why:

- **The patch is retired on current Windows installs.** Claude Desktop for Windows now ships as an MSIX package (`C:\Program Files\WindowsApps\Claude_<ver>_x64__pzs8sxrjxfjjc\app\`), which cannot and must not be modified. It also does not need to be: an unconditional block in `.vite/build/index.pre.js` reads `CLAUDE_USER_DATA_DIR` and calls `app.setPath('userData', dir)` + `setPath('logs', <dir>\Logs)` right before `requestSingleInstanceLock()`, so `Cmd-Open` sets that env var per profile (via `Start-ClaudeInstance`) and simultaneous side-by-side logins just work — verified live on `1.21459.1.0` (own `Network\Cookies`; `Logs\main.log` appearing inside the profile dir is the proof the env block ran). App resolution (`Find-AppDir`) tries `Get-AppxPackage -Name Claude` first and falls back to legacy Squirrel `%LOCALAPPDATA%\AnthropicClaude\app-*`; `patch` dies with an explainer on MSIX and only ever runs against a Squirrel target. `--profile=<name>` is still passed at launch — inert in the app, but `Profile-Running` identifies instances by scanning process args for it (main-process detection also filters on `ExecutablePath`, because the Claude Code CLI is *also* a `claude.exe` with no `--type=` and would otherwise read as "default running" forever). Accepted losses vs. the patch: no `[name]` window-title prefix, no automatic sessionKey export for the dashboard. Two launch gotchas, both learned the hard way: a terminal hosted inside Claude Desktop leaks `CHROME_CRASHPAD_PIPE_NAME`, which makes a spawned Claude misbehave as a crashed child process (`Start-ClaudeInstance` scrubs `CHROME_*`/`ELECTRON_*` first), and profiles must never be launched via `explorer.exe`/`shell:AppsFolder` (env vars do not pass through Explorer). If Claude ever fails to start with "Another program is currently using this file" after an update: stale Appx registrations (some stuck in `Servicing` state) — from admin PowerShell, `Get-AppxPackage -AllUsers -Name Claude`, `Remove-AppxPackage -AllUsers` everything but the newest, reboot once.

Everything below describes the legacy Squirrel-era patch flow, kept for machines still on the old installer:

- **The injected JS must stay byte-identical between `_write_injector` (sh heredoc) and `Write-Injector` (ps1 here-string).** The JS itself is platform-aware where behavior must differ: `linkDir()` creates a `'junction'` on win32 (works with no admin rights and no Developer Mode) and a plain symlink elsewhere. `Write-Injector` normalizes CRLF back to LF (a Windows checkout with autocrlf would otherwise embed CRLF) and re-appends the trailing newline that PowerShell here-strings drop. When touching the injection, edit both copies and diff the extracted blocks.
- **No Info.plist, no codesign on Windows.** Only `app.asar` is modified, never `claude.exe`, so Authenticode is untouched. The launch-safety risk is instead Electron's `EnableEmbeddedAsarIntegrityValidation` fuse: on Windows the expected asar-header hash is embedded in the signed exe and cannot be safely rewritten. `Assert-AsarIntegrityNotEnforced` reads the fuse wire (32-byte sentinel `dL7pKGdnNz796PbbjQWNKmHXBZaB9tsX`, then 1 version byte, 1 length byte, one state byte per fuse; the asar fuse is #5, 1-based; layout documented in the generated `fuse-check.js`) and refuses to patch, touching nothing, if the fuse is on or the wire is unrecognizable. The version and length are RAW bytes (`0x01`, not ASCII `'1'`) while only the fuse-state bytes are ASCII — `fuse-check.js` reads the version as a number accordingly (reading it as a char printed an invisible control byte and a baffling "unknown version ('')"). This gate only matters for the legacy Squirrel patch path; the last Squirrel builds shipped the fuse ON, which (together with the move to MSIX) is why the patch is retired on Windows.
- **App target:** newest `%LOCALAPPDATA%\AnthropicClaude\app-<version>\` (Squirrel installs each version as a fresh dir, which is also why updates silently unpatch; backup staleness is keyed on the version string like on macOS). Everything is user-writable, so no elevation anywhere. App resolution is lazy (`Require-AppPaths`) so `install`/`help`/`dash`/`list`/`doctor` work on a machine without Claude.
- **Zero-unpacked originals repack with NO unpack rule** (`Pack-PreservingUnpacked`): the generic fallback glob would wrongly unpack files the original kept packed and fail the set-equality gate. The exact-basename primary and generic fallback (`{**/*.node,**/*.dll,**/*.exe,...}`) mirror the macOS logic; a single basename is passed bare because minimatch does not expand one-element braces. No executable-bit restore: NTFS has none to lose.
- **Windows PowerShell 5.1 compatibility:** no PS7-only syntax (`&&`, ternary), `Set-StrictMode 2`. Native commands with `2>$null` go through `Invoke-NativeQuiet`, because 5.1 turns redirected native stderr into terminating errors under `$ErrorActionPreference = 'Stop'`. Node helper snippets are written as files into the tool dir, not passed via `node -e` (5.1 mangles multi-line native args). The `@electron/asar@^3` pin lives inside `package.json`, not on the npm command line, because cmd.exe eats `^`.
- **Main-process detection:** every Electron child on Windows is also `claude.exe`; children always carry `--type=` and never `--profile=`, so filtering out `--type=` leaves one main process per instance. Used identically by `Profile-Running` (ps1) and the win32 branch of `getRunningProfiles` (server.js).
- `dashboard/server.js` branches on `process.platform`: PowerShell/CIM for process listing, `claude-deck.ps1` lookup for the Open button, and a detached `spawn` for the direct-launch fallback (an `execFile` wait-for-exit would hang the HTTP response until Claude quits).
- Session-index linking verifies the junction actually exists after `New-Item` (which can silently no-op) instead of trusting the absence of an exception. `watchdog` is not implemented on Windows; `doctor` reports when a re-patch is needed.
- **Testability on any OS:** the same `--app` override plus fake `USERPROFILE`/`LOCALAPPDATA`/`APPDATA`/`TEMP` env vars let the whole patch/revert/rollback flow run under `pwsh` on macOS against a fixture app (this is how it was smoke-tested).

## Portability constraints: don't break these

These apply to `claude-deck.sh`; the Windows script's own constraints are in the section above.

- **macOS Bash 3.2 only.** Default `/bin/bash` is 3.2 (no associative arrays, no `mapfile`, no `${var^^}`). Shebang is pinned to `/bin/bash`. `set -eu` + `set -o pipefail`.
- **macOS-native tools only** in the hot path: `curl`, `tar`, `shasum`, `codesign`, `osascript`, `PlistBuddy`, `xattr`, `launchctl`. No Homebrew. No GNU coreutils flags.
- **Node bootstrap.** If `node` is missing or <18, the script downloads Node 20 LTS (Apple Silicon or Intel, detected via `uname -m`) into `~/.claude-deck/`, never system-wide. Installer preference for `@electron/asar`: `bun` → `pnpm` → bootstrapped `npm`. The dashboard itself is stdlib-only Node (>=18, uses global `fetch`): no dependency to install to run it.
- **BSD sed/awk.** Sentinel-bounded removal (zshrc alias, uninstall) uses `awk`, because BSD `sed -i` semantics differ from GNU.

## State that lives outside the repo

- `~/.claude-deck/bin/claude-deck.sh`: canonical install location (the `install` target). The `claude-deck` alias in `~/.zshrc` points here.
- `~/.claude-deck/backup/`: `app.asar.orig`, `app.asar.unpacked.orig` (if the app has unpacked native modules), `original-hash.txt`, `claude-version.txt`.
- `~/.claude-deck/backup-alt/`: same layout as `backup/`, used only when `--app`/`CLAUDE_DECK_APP` points at a non-default bundle (testing), so it can never read from or clobber the real backup.
- `~/.claude-deck/profiles/<name>.json`: one file per profile, written by the injected code: `{ name, sessionKey, orgId?, updatedAt }`, file mode 600, dir mode 700. The dashboard reads these; it never writes a session key itself.
- `~/.claude-deck/tool/`: local `@electron/asar` install.
- `~/.claude-deck/node/`: bootstrapped Node 20, only if system Node was missing or too old.
- `~/.claude-deck/cursor/accounts.json` (dashboard) and `~/.claude-deck/cursor/agent-keys.json` (delegation, mode 600): optional Cursor config, both read-only from claude-deck's side. `~/.claude-deck/bin/cursor-run.sh` + `cursor-mcp.js`: the delegation runner and its MCP wrapper, placed by `integrations/claude-code/install.sh`. See "Cursor integration".
- `~/Library/Application Support/Claude Profiles/<name>/`: per-profile Electron `userData`. The default (no `--profile`) instance is untouched and reports as profile name `default`. `claude-deck open default` and the dashboard's Open button both special-case the literal name `default` to mean "launch/focus the flag-less instance": neither ever spawns `--profile=default`.
- `~/.zshrc`: sentinel-wrapped alias block (`# >>> claude-deck shortcut >>>` … `<<<`). `uninstall` removes exactly that block and saves a `.bak.<timestamp>` first.
- `/usr/local/lib/claude-deck/`: root-owned copy of the watchdog script, installed by `watchdog on`. A user-writable script must never run as root, so `watchdog on` copies the script here (root:wheel) before wiring up the LaunchDaemon; it never points the daemon at a path the current user can edit.
- `/Library/LaunchDaemons/com.smklabs.claude-deck.plist`: installed by `watchdog on`, removed by `watchdog off`. Watches `Info.plist` for changes, skips while Claude is running, logs to `/var/log/claude-deck.log`. launchd never sets `SUDO_USER`, so `watchdog on` bakes the invoking user's name into the plist's `EnvironmentVariables` as `CLAUDE_DECK_USER`; the script re-anchors `HOME`/`USER` from that variable when it fires as root, so the daemon reads and writes the real user's `~/.claude-deck` (not root's).

## Usage normalization (`dashboard/server.js`)

The live `/api/organizations/<id>/usage` payload carries two parallel representations. `normalizeUsage` prefers the canonical `limits` array (`[{ kind, group, percent, severity, resets_at, is_active, scope: { model: { display_name } } }]`) because that is exactly what the Claude app renders: `kind` is `session` / `weekly_all` / `weekly_scoped`, and a per-model weekly limit (e.g. **Fable**) appears ONLY here as `weekly_scoped` with `scope.model.display_name`. It exists nowhere in the flat keys. The legacy flat-key scan (`five_hour`, `seven_day`, `seven_day_*`) is a fallback for older payloads without `limits`, and it deliberately skips dollar-denominated caps (objects carrying `limit_dollars`, like the `amber_ladder` credit pool) plus the null codename placeholders (`tangelo`, `iguana_necktie`, `nimbus_quill`, ...) so they never surface as bogus "Other" rows. This was a real bug: scanning flat keys showed `amber_ladder` at 0% and missed Fable at 100%. Keep the mock fixtures emitting the real dual shape (flat keys AND a `limits` array) so the primary path stays covered.

**Plan multiplier / seat tier.** Each org record carries a server-derived `multiplier` (weekly capacity relative to Pro: 20 / 6.25 / 5 / 1.25 / 1; 0 = free, null = unknown). Personal plans come from the org's `rate_limit_tier` (`default_claude_max_20x`, `default_claude_max_5x`, `default_claude_ai` = free). Team orgs all report the same tier (`default_raven`), so the seat type lives ONLY in `/api/bootstrap` → `account.memberships[].seat_tier` (`team_tier_1` = premium 6.25x, `team_standard` = 1.25x) — verified against the live API; the same bootstrap call already supplies the email, so it costs no extra request. Absolute token counts are NOT exposed (usage is percent-only), so the default "Best value" sort scores each org as `multiplier × (100 − weekly%) ÷ hours-until-weekly-reset^TIME_WEIGHT` (the exponent, currently 2.5 and tuned by feel in `index.html`, makes urgency outweigh raw capacity: a small plan resetting in hours beats a big plan resetting in days), then multiplied by a session pace factor: the session budget share left is compared to the share of the 5h session window still ahead. Budget short of time (`deficit`) projects a lockout and costs `1.2 × deficit` of the score (floored at ×0.05, never a hard disqualifier); budget ahead of time (`surplus`, capacity about to expire unused) earns a `0.5 × surplus` use-it-or-lose-it bonus; a fresh session is exactly neutral so weekly stays the primary signal. Free orgs (multiplier 0) never appear in up-next. Cursor seats whose monthly credit is ahead of pace (remaining% ÷ month-left% ≥ 1.25, assuming a 30-day cycle) get a dashed "delegate" pill appended to the same Up next row. Cached org lists in `profiles/<name>.json` written before `multiplier` existed are treated as stale and refetched.

**Untouched weeks.** Plain-language: an idle org's weekly limit only starts counting when the first message is sent, so its whole week is still up for grabs; those orgs get their own row under Up next instead of drowning at the bottom of the ranking. Detection (`orgWeekNotStarted`, `index.html`): the org has weekly windows and ALL of them are 0% with `resets_at` null — verified live; a started org always has at least one weekly window with a reset time, even when a scoped bucket (Fable) shows null. These orgs leave the main ranked row (no countdown to rank by) and render in the "Untouched weeks" row sorted by `multiplier` alone, biggest first (null = 1x convention), free orgs (multiplier 0) dropped, same active-filter gating, capped at `UP_NEXT_MAX`. Pill click is an org-targeted open, so the first message starts the week of the org actually picked. Mock fixtures cover order (Client A Team 6.25x before Research Max 5x) and the free filter (Client A Free must never appear); `resets_hrs: null` in a fixture means "window not started".

## Cursor integration (optional)

Two independent, off-by-default features that surface idle Cursor seats. Neither touches the patch flow, `Claude.app`, or any Claude profile. Plain-language: one shows Cursor usage in the dashboard, the other lets Claude Code hand tasks to Cursor.

**Dashboard adapter (`dashboard/cursor.js`, `GET /api/cursor`).** We cannot inject into Cursor (no patch), so instead of a clean `profiles/<name>.json` we read Cursor's own SQLite store. Two verified facts are load-bearing:
- `state.vscdb` is a live, WAL-mode, ~300MB file the running Cursor holds open; a plain read fails `SQLITE_CANTOPEN`. It is opened through `sqlite3 -json 'file:...?immutable=1'` (macOS ships sqlite3 3.51), which skips all locking and the WAL and returns in ~4ms with no copy. Windows has no bundled sqlite3, so token reads degrade cleanly there.
- The stored `cursorAuth/accessToken` is a ~60-day session JWT, and `refreshToken` is the SAME JWT stored twice, not a separate long-lived credential. So there is no safe way to mint a fresh token; expiry is checked locally from the JWT `exp` and an expired account shows "open Cursor once to refresh", mirroring the Claude expired-key UX. **We never POST the refresh token anywhere, so we can never log the account out of Cursor.** Disk data (label/email/plan from `stripeMembershipType`/token validity) is always reliable; live usage (`cursor.com/api/usage?user=<id>` with the `WorkosCursorSessionToken={userId}::{token}` cookie, userId = JWT `sub` after `auth0|`) is best-effort and must never break a card. `normalizeCursorUsage` is defensive (legacy per-model request shape AND credit/dollar shape) and computes cycle resets with UTC date methods, not local (a real bug the tests caught). Auto-discovers the default install when no `accounts.json`; Admin API path (Enterprise, basic auth) is lean and opt-in. Cached 120s like `/api/usage`; mock fixtures cover every card state.

**Delegation (`integrations/claude-code/`).** Agent calling, NOT a model-backend swap: Cursor sells no Anthropic-shaped API for its subscription and the Desktop Code tab always uses the claude.ai login, so Claude's engine cannot point at Cursor. Instead `cursor-agent` runs alongside as a worker. `cursor-run.sh` is the single source of truth for invoking it (the skill, the `cursor-worker` subagent, and the `cursor-mcp.js` MCP tool all shell out to it, mirroring the injected-JS/shell-twin belt-and-suspenders pattern). Auth per `--account <label>`: an API key from `agent-keys.json` wins, else a login slot at `~/CursorProfiles/<label>/cli-home` (cursor-agent keeps its credential in `$HOME/.cursor/cli-config.json`; verified that `CURSOR_CONFIG_DIR` does NOT move it but a `HOME` override does, which is the whole slot mechanism); no account at all = ambient base login. Never the fragile session token the dashboard reads. `--dry-run` redacts keys; `--trust` is always passed because headless runs die on the interactive trust prompt. Bash 3.2, `node` only for the JSON key lookup. cursor-agent reads `CLAUDE.md` directly, so instructions are shared for free; Cursor "Auto" is unlimited on paid plans while named models draw the pool. When touching the injected JS twins keep them behaviorally identical, same rule here for `cursor-run.sh` and its three callers.

**`server.js` PORT.** `process.env.PORT` was added as the lowest-precedence port fallback (after `CLAUDE_DECK_PORT`, then the positional arg, then `8965`) so PaaS hosts and preview harnesses can assign a port; it never overrides an explicit choice.

## Shared account data, separate accounts

Every profile launches the same patched app copy and points at the same `~/.claude`, so Claude Code transcripts, config, and settings are identical across every logged-in account. Only the Electron `userData` (chat history, cookies, local storage: the account login itself) is per-profile. This is the entire point: one 700MB app bundle, unlimited simultaneous logins, no duplicated installs.

The one thing that used to be per-`userData`, and isn't anymore, is the Claude Code **session index** (as opposed to the transcripts themselves): see step 4's "Shares one Claude Code session index across profiles" above. Without that symlink, a second profile of the same account has its own empty index and shows no Code sessions even though the shared transcripts are sitting right there in `~/.claude/projects`. The symlink is what keeps this section's "identical across every logged-in account" claim actually true. `claude-sync` scans only the default app's single `claude-code-sessions` directory, so this symlink is also what keeps its single-directory assumption valid; seeing sessions from a genuinely *different* account still requires running `claude-sync` itself.

## Migrating profiles to another machine

Move the small profile JSONs, not the profile folders: the login travels in the JSON, and the injector plants it back on first launch.

- A raw `userData` copy cannot carry a login. Electron encrypts cookies at rest with an OS-bound key (macOS Keychain / Windows DPAPI), so a copied cookie store is undecryptable on the new machine and the profile opens logged out.
- Path: copy `~/.claude-deck/profiles/*.json` into the target machine's `~/.claude-deck/profiles/` (Windows: `%USERPROFILE%\.claude-deck\profiles\`), patch the app there, launch each profile normally.
- The injector's `whenReady` handler (step 4) seeds the JSON's `sessionKey` as the `claude.ai` cookie only when the session has none, then reloads any window that loaded before the seed landed. A live login always wins; every step silent-fails. No command needed: seeding is automatic.
- These JSONs are full account credentials (mode 600). Move them like passwords and delete transfer copies.
