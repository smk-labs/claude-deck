# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Never run the patch without asking

**Do not run `claude-deck patch`, `claude-deck revert`, or anything that touches `/Applications/Claude.app` unless the user explicitly asks for it in this session, and only after confirming Claude Desktop is quit.** Smoke-testing the patch quits the running app: `patch` and `revert` both need Claude closed to rewrite `app.asar`, and the script will try to quit it via `osascript`. If Claude is open with unsaved context (an in-progress chat, a login flow), killing it is disruptive. Ask first, every time. `status` and `list` are read-only and always safe to run.

## What this repo is

A single Bash script (`claude-deck.sh`) plus a small local dashboard (`dashboard/`). The script patches the macOS Claude Desktop app to accept a `--profile=NAME` launch argument, so each profile gets its own Electron `userData` directory, its own login session, and can run side by side with the others. The dashboard is a zero-dependency Node HTTP server that reads each profile's session key and shows its usage limits. There is no build system, no package manager, no tests: the script and the two dashboard files *are* the product.

## Running it

```bash
./claude-deck.sh patch              # apply (idempotent)
./claude-deck.sh patch --force      # re-apply even if already patched
./claude-deck.sh patch --verify-launch  # (only with --app <scratch-copy>) launch + confirm it stays alive 8s
./claude-deck.sh revert             # restore original Claude.app from backup
./claude-deck.sh status             # show patch state, hashes, backup info, known profiles
./claude-deck.sh open <name>         # launch (or focus) Claude with --profile=<name>
./claude-deck.sh list                # list known profiles and their cached usage
./claude-deck.sh dash [port]         # start the local dashboard (default port 8965)
./claude-deck.sh install             # copy to ~/.claude-deck/ + add zsh alias
./claude-deck.sh uninstall           # remove the alias only
./claude-deck.sh watchdog on|off     # install/remove the LaunchDaemon that re-patches on update (needs sudo)
```

`patch` and `revert` need `sudo` (writes into `/Applications/Claude.app`) and need Claude quit first. `open`, `list`, `dash`, and `status` never touch the app bundle and never need sudo.

To smoke-test a change: confirm with the user, then edit the script, run `./claude-deck.sh revert` (if previously applied), then `./claude-deck.sh patch`, then `./claude-deck.sh open work` and verify the window title shows `[work]` and the profile gets its own login. `status` is the fastest sanity check between iterations.

## Architecture / how the patch works

The things that make this work as a self-contained script: touch any of them and you can break the whole flow:

1. **Asar extract → inject → repack.** The Electron app's code lives in `/Applications/Claude.app/Contents/Resources/app.asar`. The script uses `@electron/asar` (installed locally into `~/.claude-deck/tool/`, **pinned to ^3**: v4+ is ESM-only and breaks both the CLI invocation and the inline `require()` in the header-hash step) to extract, write `claude-deck.js` at the asar root, prepend `try { require('../../claude-deck.js') } catch (e) {}` to `.vite/build/index.pre.js` (the Electron main-process entry point), then repack.

1a. **Unpacked native modules must survive the repack, or the app never opens.** Modern Claude Desktop ships native `.node`/`.dylib` files and node-pty's `spawn-helper` marked `unpacked:true` in the asar header (materialized on disk in `Contents/Resources/app.asar.unpacked`), because Electron cannot `dlopen`/`execve` a file that lives inside the asar archive. A plain `asar pack` with no `--unpack` rule silently repacks everything, and the main process crashes before any window opens: this bricked the app once already. `asar_pack_preserving_unpacked()` collects the original unpacked file list, packs with an exact basename brace-glob (`@electron/asar` matches `unpack` against each file's basename, not its full path, so full-path globs match nothing), falls back to a generic `{**/*.node,**/*.dylib,**/spawn-helper}` pattern if that doesn't reproduce the set, and the caller re-verifies set equality against the original before installing anything. `asar.extractAll` also drops the executable bit on unpacked files (they land as plain 644), so `cmd_patch` restores each one's real mode from the live `Resources/app.asar.unpacked` before repacking, or `spawn-helper` (needs `+x` to `execve`) would silently stop working even though the app would still launch.

2. **ElectronAsarIntegrity hash.** Electron refuses to load the asar if the SHA-256 in `Info.plist` (`:ElectronAsarIntegrity:Resources/app.asar:hash`) doesn't match. The hash is computed over the asar **header JSON**, not the whole file. After repack, the script recomputes it and updates the plist via `PlistBuddy`.

3. **Ad-hoc re-sign, never `--deep`, entitlements stripped not preserved.** `codesign --force --sign - --preserve-metadata=identifier,flags,runtime --entitlements <derived.plist>` on the outer bundle only. **Do not add `--deep`**: it re-signs nested helpers as ad-hoc, which invalidates their keychain ACLs and causes repeated keychain prompts on every Claude launch. This is one of two load-bearing constraints in this mechanism; the fix carries over verbatim from the sibling `claude-rtl` project.
   - **Why entitlements are no longer just preserved.** The real install is hardened-runtime and carries Apple *restricted* entitlements (`com.apple.application-identifier`, `com.apple.developer.team-identifier`, `keychain-access-groups`), which AMFI only permits under a genuine Apple team signature. `--preserve-metadata=...,entitlements,...` used to carry those onto the ad-hoc signature verbatim; on macOS 26 (Tahoe) AMFI then refuses to launch the binary at all ("adhoc signed but contains restricted entitlements") even though `codesign --verify` reports the signature as structurally valid. Dropping those keys then breaks Library Validation for the still-genuinely-signed nested Electron Framework ("different Team IDs"), so `com.apple.security.cs.disable-library-validation` is added to compensate. Hardened runtime stays on; `--deep` is still never used (disable-library-validation makes it unnecessary).
   - **`build_adhoc_entitlements()`** derives this plist fresh from the target app's own current entitlements every time (dump via `codesign -d --entitlements`, handling both the direct-file and `:-`-stdout forms; strip the three restricted keys with `PlistBuddy -c Delete`; add the disable-library-validation key), so it self-adapts if Anthropic's entitlement set changes later. Falls back to a hardcoded minimal set (JIT + the device/personal-information/virtualization keys the app is known to use) only if the app is unsigned and the dump comes back empty. Used identically by `cmd_patch`'s success path, its rollback path, and `cmd_revert`: revert can only ever produce an ad-hoc signature too (the pristine asar's original signature was Anthropic-genuine, but ad-hoc can't carry it), so it needs the same stripped set.
   - **`assert_launch_safe_entitlements()`** is a static post-sign gate: dumps the just-signed entitlements back and asserts none of the three restricted keys survived and `disable-library-validation` is `true`. This exists because `codesign --verify` alone is a false-negative here: it validates signature structure, not AMFI's separate launch-time policy check, so it happily passed on a binary AMFI still refused to run. Runs in `cmd_patch` (fails → rollback trap fires) and `cmd_revert` (fails → `die`).

4. **The injection itself** (`claude-deck.js`, written at patch time). Runs in the main process before `app.ready`, wrapped so nothing inside it can crash the app:
   - Reads `--profile=` from `process.argv`, sanitizes to `[A-Za-z0-9_-]`, caps at 32 chars.
   - If a profile is set, calls `app.setPath('userData', ...)` (and best-effort `app.setPath('sessionData', ...)` in its own try/catch) pointing at `~/Library/Application Support/Claude Profiles/<name>`.
   - **Shares one Claude Code session index across profiles.** Claude Desktop keeps its Claude Code session list per-`userData`, at `<userData>/claude-code-sessions/<account-uuid>/<org-uuid>/local_*.json`. Transcripts live in the shared `~/.claude/projects`, but the app only lists sessions found in this per-profile index, so a second profile of the same account showed "no Code sessions" even with transcripts present. Fix: symlink each profile's `claude-code-sessions` dir at the default app's own (`~/Library/Application Support/Claude/claude-code-sessions`), so every profile reads and writes the same index. If a profile already has a real (non-symlink) index dir, its contents are migrated into the shared dir additively (never overwriting a file already there), the original directory is kept as a `.migrated-<timestamp>` backup (never deleted), and only then is the symlink created. Runs once at require time, before `app.on('ready')`, since the fix must be in place before the app first touches the dir. Every step has its own try/catch; any failure leaves the old per-profile behavior in place rather than blocking launch. This is also why the shared `~/.claude` design keeps `claude-sync`'s single-directory scan assumption valid: see "Shared account data, separate accounts" below. The same logic also exists as a shell twin, `ensure_profile_index_link()`, run by `cmd_open` before every profile launch, by `cmd_dash` at startup, and by `cmd_doctor` for all profiles at once. That makes linking self-healing even when the installed app carries an outdated injection (a real failure mode, hit twice: a scoping bug in the injected JS, then an un-repatched app). The injected copy is belt, the shell twin is suspenders; keep them behaviorally identical when touching either.
   - On `app.on('browser-window-created')`, listens for `page-title-updated`, calls `preventDefault()`, and sets the title to `[<profile>] <original title>` so Mission Control, Cmd-backtick, and Raycast can tell windows apart.
   - After `app.whenReady()`, reads the `sessionKey` cookie for `https://claude.ai` and writes it to `~/.claude-deck/profiles/<label>.json` (mode 600), merging into any existing file so a cached `orgId` survives. Re-pulls every 30 minutes and on `ses.cookies.on('changed')` for `sessionKey`. Every failure path is silent: this code must never be the reason Claude fails to launch.

5. **Backup + idempotency.** First patch copies the pristine asar (and `app.asar.unpacked`, if present) to `~/.claude-deck/backup/` and stashes the original Info.plist hash + Claude version. The marker `claude-deck.js` inside the asar is the "is patched" signal. `revert` restores the asar, the unpacked dir, and the hash, then re-signs. `patch --force` re-applies even when the marker is already present (useful after editing the injected script).

6. **Preflight gate, rollback trap, post-validation.** `cmd_patch` checks `.vite/build/index.pre.js` exists in the asar *before* touching anything (Claude's internal layout can change between versions); if it's gone, nothing is modified. From the moment the installed app starts changing until success, an `EXIT`/`INT`/`TERM` trap (`_patch_rollback`) is armed: any failure restores the backup asar, the backup unpacked dir, and the plist hash, re-signs, and reports honestly if the restore itself couldn't fully complete (e.g. a read-only `Info.plist`) rather than claiming false success. `PlistBuddy -c "Set ..."` can exit 0 while silently failing to write, so the trap doesn't trust its own exit code either: it re-reads the plist afterward and compares. Before printing success, a post-validation gate re-derives ground truth (asar header hash equals the plist's, the marker is present, the unpacked set still matches, `codesign --verify` passes) and only then clears the trap.

7. **Testability via `--app <path>` / `CLAUDE_DECK_APP`.** Every subcommand accepts a hidden `--app` override (or the env var) that redirects `APP`/`RES`/`ASAR`/`PLIST` at an alternate `.app` bundle, so the whole patch/revert/rollback flow can be smoke-tested against a scratch copy without touching the real `/Applications/Claude.app`. When the target bundle is user-writable, every file mutation and `codesign` call skips `sudo` automatically (`$SUDO` resolves to `""`). Backups for an alternate app live under `~/.claude-deck/backup-alt/`, never `~/.claude-deck/backup/`, so a test run can never read from or clobber the real backup. `quit_claude()` itself is scoped the same way: it's a no-op unless `$APP` is exactly `/Applications/Claude.app`, since `osascript`/`pkill` there target the process by bare name ("Claude") and would otherwise hit the real running app regardless of which bundle is actually being patched. `patch --verify-launch` builds on this: it spawns the freshly-patched `$APP` with a throwaway `--user-data-dir` and confirms the process is still alive 8 seconds later (the one check that exercises the real AMFI launch-time policy end-to-end, vs. `assert_launch_safe_entitlements`'s static proxy for it). It refuses outright, via `die`, if `$APP` is the real Claude.app.

## Portability constraints: don't break these

- **macOS Bash 3.2 only.** Default `/bin/bash` is 3.2 (no associative arrays, no `mapfile`, no `${var^^}`). Shebang is pinned to `/bin/bash`. `set -eu` + `set -o pipefail`.
- **macOS-native tools only** in the hot path: `curl`, `tar`, `shasum`, `codesign`, `osascript`, `PlistBuddy`, `xattr`, `launchctl`. No Homebrew. No GNU coreutils flags.
- **Node bootstrap.** If `node` is missing or <18, the script downloads Node 20 LTS (Apple Silicon or Intel, detected via `uname -m`) into `~/.claude-deck/`, never system-wide. Installer preference for `@electron/asar`: `bun` → `pnpm` → bootstrapped `npm`. The dashboard itself is stdlib-only Node (>=18, uses global `fetch`): no dependency to install to run it.
- **BSD sed/awk.** Sentinel-bounded removal (zshrc alias, uninstall) uses `awk`, because BSD `sed -i` semantics differ from GNU.

## State that lives outside the repo

- `~/.claude-deck/bin/claude-deck.sh`: canonical install location (the `install` target). The `claude-deck` alias in `~/.zshrc` points here.
- `~/.claude-deck/backup/`: `app.asar.orig`, `app.asar.unpacked.orig` (if the app has unpacked native modules), `original-hash.txt`, `claude-version.txt`.
- `~/.claude-deck/backup-alt/`: same layout as `backup/`, used only when `--app`/`CLAUDE_DECK_APP` points at a non-default bundle (testing), so it can never read from or clobber the real backup.
- `~/.claude-deck/profiles/<name>.json`: one file per profile, written by the injected code: `{ name, sessionKey, orgId?, updatedAt }`, file mode 600, dir mode 700. The dashboard reads these; it never writes a session key itself.
- `~/.claude-deck/tool/`: local `@electron/asar` install.
- `~/.claude-deck/node/`: bootstrapped Node 20, only if system Node was missing or too old.
- `~/Library/Application Support/Claude Profiles/<name>/`: per-profile Electron `userData`. The default (no `--profile`) instance is untouched and reports as profile name `default`. `claude-deck open default` and the dashboard's Open button both special-case the literal name `default` to mean "launch/focus the flag-less instance": neither ever spawns `--profile=default`.
- `~/.zshrc`: sentinel-wrapped alias block (`# >>> claude-deck shortcut >>>` … `<<<`). `uninstall` removes exactly that block and saves a `.bak.<timestamp>` first.
- `/usr/local/lib/claude-deck/`: root-owned copy of the watchdog script, installed by `watchdog on`. A user-writable script must never run as root, so `watchdog on` copies the script here (root:wheel) before wiring up the LaunchDaemon; it never points the daemon at a path the current user can edit.
- `/Library/LaunchDaemons/com.smklabs.claude-deck.plist`: installed by `watchdog on`, removed by `watchdog off`. Watches `Info.plist` for changes, skips while Claude is running, logs to `/var/log/claude-deck.log`. launchd never sets `SUDO_USER`, so `watchdog on` bakes the invoking user's name into the plist's `EnvironmentVariables` as `CLAUDE_DECK_USER`; the script re-anchors `HOME`/`USER` from that variable when it fires as root, so the daemon reads and writes the real user's `~/.claude-deck` (not root's).

## Shared account data, separate accounts

Every profile launches the same patched app copy and points at the same `~/.claude`, so Claude Code transcripts, config, and settings are identical across every logged-in account. Only the Electron `userData` (chat history, cookies, local storage: the account login itself) is per-profile. This is the entire point: one 700MB app bundle, unlimited simultaneous logins, no duplicated installs.

The one thing that used to be per-`userData`, and isn't anymore, is the Claude Code **session index** (as opposed to the transcripts themselves): see step 4's "Shares one Claude Code session index across profiles" above. Without that symlink, a second profile of the same account has its own empty index and shows no Code sessions even though the shared transcripts are sitting right there in `~/.claude/projects`. The symlink is what keeps this section's "identical across every logged-in account" claim actually true. `claude-sync` scans only the default app's single `claude-code-sessions` directory, so this symlink is also what keeps its single-directory assumption valid; seeing sessions from a genuinely *different* account still requires running `claude-sync` itself.
